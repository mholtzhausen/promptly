# Repository Guidelines

## Project Overview

**Prompt Tray** (`prompt_tray`) — a Linux desktop system-tray application written in Rust + GTK4 that manages prompt templates with variable placeholders. It stores templates in SQLite, provides fuzzy-search selection, auto-generates type-aware input dialogs for template variables, and copies the final interpolated prompt to the clipboard.

## Architecture & Data Flow

```
┌──────────┐    mpsc       ┌─────────────────┐
│ rdev     │ ────(tx/rx)──▶│ GTK4 Main Loop   │
│ hotkey   │  50ms poll     │ (app.run)        │
│ thread   │                │ ┌─────────────┐  │
└──────────┘                │ │ PopupWindow  │  │
                            │ │ (ListBox +   │  │
┌──────────┐                │ │  SearchEntry)│  │
│ ksni     │                │ └──────┬──────┘  │
│ Tray     │◀───send───────│        │          │
│ icon     │    "show"     │   ┌────▼──────┐   │
└──────────┘                │   │ Variable  │   │
                            │   │ Dialog    │   │
                            │   │ (live     │   │
                            │   │  preview) │   │
                            │   └────┬──────┘   │
                            │        │          │
                            │   ┌────▼──────┐   │
                            │   │ (GDK)     │
                            └─────────────────┘
```

- **No async/await** — fully synchronous GTK4 + blocking I/O. The only event loop is `app.run()`.
- **Hotkey detection**: `rdev::listen()` in a spawned `std::thread`. Sends `()` through an `mpsc::Sender` to a 50ms polling loop in the GTK main thread (`glib::timeout_add_local`).
- **System tray**: `ksni` crate (freedesktop StatusNotifierItem protocol, works on X11 + Wayland). Created before `app.run()`; lives in a background thread managed by the crate.
- **Shared popup state**: `Arc<Mutex<Option<PopupWindow>>>` — `None` means hidden, `Some` means visible. Toggled on hotkey.
- **Per-operation DB connections**: each dialog open/close creates its own `rusqlite::Connection`. No connection pooling.
- **Window hints**: X11 calls (`XMoveWindow`, `_NET_WM_STATE_ABOVE`) via FFI through `x11` crate. Graceful no-op on Wayland.

## Runtime / Tooling Preferences

- **Linux only** — X11 required for window hints (centering, always-on-top). Wayland supported for GUI but X11 features are no-ops.
- **Rust edition 2021**, stable toolchain.
- **GTK4** system library (`libgtk-4-dev` for building, `libgtk-4-1` for runtime).
- **libxdo3** system library — accessed via FFI through a local `./lib/libxdo.so` symlink (created by `make libxdo`).
- **Release profile**: `opt-level=2`, LTO enabled.
- **Package manager**: Cargo only. No npm, pip, etc.
- **No async runtime** — synchronous code, blocking I/O, `std::thread::spawn` for hotkey listener.

## Code Conventions & Common Patterns

### Formatting
- Standard Rust formatting (rustfmt assumed; no config file present).
- `anyhow::Result` for error-propagating functions.

### Naming
- `snake_case` for functions, variables, modules.
- `PascalCase` for structs, enums, type aliases.
- `SCREAMING_SNAKE_CASE` for constants (`APP_NAME`, `PLACEHOLDER_RE`).
- File names match the module they define (one module per file, no mod.rs).

### Error Handling
- `anyhow::Result<T>` everywhere — no custom error types.
- DB failures in hotkey handler: `log::error!` + `std::process::exit(1)`.
- Hotkey listener errors: `log::warn!` — app continues.
- **No-variable copy path**: `gdk::Display::default()` → `None` logs error and shows user-facing notification (`"Prompt not copied"`). Variable dialog copy uses GDK clipboard via `WidgetExt::display()` — no error branch needed (`gdk4::Clipboard::set_text` returns `()` in 0.8.2).

### State Management
- Shared mutable state: `Rc<RefCell<T>>` for single-threaded GTK closures (e.g., `Rc<RefCell<Vec<Prompt>>>` inside `PopupWindow`).
- Cross-thread shared state: `Arc<Mutex<T>>` (e.g., `Arc<Mutex<Option<PopupWindow>>>` for popup lifecycle).
- Callback storage in structs: `Rc<dyn Fn(...)>` type aliases.

### GTK4 Patterns
- **Builder pattern**: `.builder()` everywhere — `ApplicationWindow::builder()`, `Dialog::builder()`, `Entry::builder()`, etc.
- **CSS**: single embedded `&str` in `config.rs` loaded via `CssProvider::load_from_data` with `STYLE_PROVIDER_PRIORITY_APPLICATION`.
- **Signal handlers**: `connect_*` methods with closures. Shared state across closures via `Rc<>`.
- **Widget extraction from VarType**: `downcast_ref::<T>()` on `gtk4::Widget` to get values from type-aware inputs.
- **No GTK Application lifecycle** — windows are standalone `ApplicationWindow` instances.

### Template Syntax
```
{{variable_name|type|default value|description}}
```
Types: `Text` (Entry), `Number` (SpinButton), `Option` (ComboBoxText with comma-separated values from the crate author), `Multiline` (TextView).

### Fuzzy Search
Custom substring-order matching in `popup.rs` — no external crate. Checks that query characters appear in order in the target string (case-insensitive).

## Testing & QA

```bash
make test   # runs cargo test with libxdo RUSTFLAGS
```

- **Test framework**: Rust's built-in `#[cfg(test)] mod tests` + `#[test]`.
- **Test files**: inline in source modules (`db.rs` has 5 tests, `prompt_parser.rs` has 10 tests).
- **Test dependencies**: `tempfile` 3 (dev-dependency) for isolated SQLite database fixtures.
- **No integration tests** — no `tests/` directory.
- **No coverage configuration** — no `codecov`, no `tarpaulin`, no CI config.
- **No linting configuration** — no `clippy.toml`, no `.rustfmt.toml`.
